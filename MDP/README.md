지금까지 원본 코드의 문제점 분석부터 시작해 수학적으로 완전하면서도 현실을 반영한 **완전한 MDP(Markov Decision Process)** 모델로 발전시킨 과정을 한눈에 보기 쉽게 마크다운으로 정리했습니다.

---

## 1. 기존 코드의 문제점 분석 (왜 완전한 MDP가 아니었는가?)

원본 코드는 시스템의 실제 물리적 상태를 은닉(Hidden)하고 에이전트에게 요약된 정보만 전달했기 때문에 MDP가 아닌 POMDP(부분 관측 가능한 MDP)에 가까웠습니다.

- **상태 추상화로 인한 마르코프 성질 위배:** 물리 큐 크기(0~50)를 에어전트에게는 4단계 레벨(0~3)로 압축해 보여주었습니다. 이로 인해 동일한 레벨이더라도 내부 실제 큐 크기에 따라 다음 상태로 갈 확률이 달라져 마르코프 성질이 깨졌습니다.
    
- **이웃 노드 정보의 블라인드 스팟:** 6개의 이웃 중 하나를 골라 오프로딩할 수 있으면서 상태 공간에는 '가장 상태가 좋은 이웃 1개'의 정보만 있었습니다. 다른 이웃을 선택했을 때의 결과를 현재 상태에서 예측할 수 없었습니다.
    
- **DP(Dynamic Programming) 적용 불가:** 전이 확률 행렬 $P(s' \mid s, a)$와 기대 보상 함수 $R(s, a)$를 수학적으로 정의할 수 없는 Model-free 시뮬레이션 구조였습니다.
    

---

## 2. 모델 재설계 전략 (차원의 저주 극복)

수십 조 개의 상태 공간을 DP 연산이 가능한 **수천 개 수준**으로 줄이기 위해 다음과 같은 통폐합 및 다이어트를 진행했습니다.

- **행동 공간의 이진화 및 구체화:** 이웃 6개를 모두 고르는 대신, 신호가 가장 잘 잡히는 상태가 좋은 이웃 노드 2개(Neighbor 1, 2)로 대상을 좁혔습니다.
    
- **환경 변수의 통합:** 채널 상태(3)와 대역폭(2)을 각각 추적하지 않고, 전송 속도에 직결되는 `Comm_State` (3단계) 변수 하나로 통합했습니다.
    
- **큐 크기의 마르코프 상태화:** 압축된 레벨을 버리고, **실제 물리 큐 크기의 정수 값 자체를 상태 공간에 직접 편입**시켜 은닉 상태를 완전히 제거했습니다.

## 3. 큐(Queue) 크기 설정의 비판적 검토 및 개선

수학적 연산량과 실제 물리적인 통신 환경(Real-world)의 균형을 맞추기 위해 초기 수치를 현실적으로 재조정했습니다.

### 기존 설정의 모순점

- 성능이 떨어지는 로컬 큐는 15개까지 버티는데, 성능이 수십 배 강력한 에지 서버(이웃 큐)는 고작 5개만 쌓여도 가득 찬다는 설정은 현실과 정반대였습니다.
    
- 이웃 큐가 너무 작으면 에이전트가 오프로딩 액션을 아예 기피하는 왜곡된 정책이 도출됩니다.
    

### 최종 개선안 (자원 스케일 다이어트)

|**구분**|**기존 설정**|**개선 설정**|**현실적인 이유**|
|---|---|---|---|
|**로컬 큐 (Local Q)**|0 ~ 15 (16단계)|**0 ~ 5 (6단계)**|단말기는 자원이 부족하므로 즉시 드롭하거나 넘겨야 함|
|**이웃 큐 (Neighbor Q)**|0 ~ 5 (6단계)|**0 ~ 10 (11단계)**|에지 서버는 대용량 처리가 가능하므로 더 큰 버퍼 수용 가능|
|**처리 속도 ($f$)**|로컬 = 이웃|**이웃이 10배 빠름**|서버의 빠른 연산 속도를 반영하여 오프로딩 유인(Incentive) 제공|

## 4. 최종 정의된 완전한(Strict) MDP 모델 Specification

이제 에이전트가 보는 상태가 시스템의 실제 물리 상태와 100% 일치하므로, 과거의 히스토리 없이 **현재 상태와 행동만으로 다음 상태와 보상을 완벽히 예측할 수 있습니다.**

### ① 행동 공간 (Action Space) : 4차원

- `0`: 로컬 처리 (Local Processing)
    
- `1`: 이웃 노드 1로 오프로딩 (Offloading to Neighbor 1)
    
- `2`: 이웃 노드 2로 오프로딩 (Offloading to Neighbor 2)
    
- `3`: 의도적 태스크 드롭 (Intentional Drop)
    

### ② 상태 공간 (State Space) 구성 변수

1. `Task_Type` : 2단계 (0: 고순위, 1: 일반) $\rightarrow$ _i.i.d. 구조로 마르코프 성질 만족_
    
2. `Comm_State` : 3단계 (0: 불량, 1: 보통, 2: 좋음) $\rightarrow$ _Markov Chain 구조로 만족_
    
3. `Local_Queue` : 6단계 (0 ~ 5 정수) $\rightarrow$ _은닉 상태 없음_
    
4. `Neighbor_1_Queue` : 11단계 (0 ~ 10 정수) $\rightarrow$ _은닉 상태 없음_
    
5. `Neighbor_2_Queue` : 11단계 (0 ~ 10 정수) $\rightarrow$ _은닉 상태 없음_
    

### ③ 총 상태 공간 개수 계산

$$\text{Total States} = 2 \times 3 \times 6 \times 11 \times 11 = 4,356\text{개}$$

> 단 **4,356개**의 상태만 존재하므로, 메모리에 전이 확률 행렬을 완전히 올릴 수 있으며 Value Iteration(DP) 알고리즘을 수초 내에 구동할 수 있습니다.

## 5. 보상 함수(Reward) 이론 및 수식

모바일 에지 컴퓨팅(MEC) 연구에서 가장 표준적으로 사용되는 비용 최소화 가중치 합산 모델(Weighted Sum Cost Model)을 적용합니다.

$$R(s, a) = - \big( W_{delay} \cdot C_{delay}(s,a) + W_{energy} \cdot C_{energy}(s,a) + C_{drop}(s,a) \big)$$

- **지연 시간 비용 ($C_{delay}$):**
    
    - 로컬 처리 시: $\frac{Q_{local} + 1}{f_{local}}$ (분모가 작아 지연 시간 큼)
        
    - 오프로딩 시: $D_{trans}(Comm) + \frac{Q_{neighbor} + 1}{f_{neighbor}}$ (분모가 커서 컴퓨팅 지연 시간은 작으나 전송 지연 발생)
        
    - 태스크 종류가 고순위(`Task_Type=0`)일 경우 지연 가중치($W_{delay}$)를 크게 부여.
        
- **에너지 비용 ($C_{energy}$):** 로컬 처리와 오프로딩(데이터 전송) 시 발생하는 물리적 배터리 소모량 반영.
    
- **드롭 페널티 ($C_{drop}$):** 의도적 드롭이나 큐가 가득 차서 발생하는 강제 버퍼 오버플로우 시 시스템 안정성을 위해 큰 음수 페널티 부여.
