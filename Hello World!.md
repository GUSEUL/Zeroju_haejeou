# 마이크로 데이터센터(MDC) 기반 통신망 오프로딩 강화학습 모델 설계도

---

## 1. 프로젝트 개요 및 환경 설정

* **최적화 목표:** 5G/6G 기지국에 배치된 마이크로 데이터센터(MDC)의 한정된 자원을 활용하여, 트래픽을 분산하고(Load Balancing) 초저지연 통신을 달성하는 태스크 오프로딩(Task Offloading) 정책 학습.
* **학습 환경 특성:** * 24시간 끊임없이 작동하는 시스템이므로 종료 상태(Terminal State)가 없는 **연속적 태스크(Continuing Task)**로 정의.
  * 보상의 무한 발산을 막기 위해 반드시 **할인율($\gamma$)**을 적용(예: 0.9 ~ 0.99).
* **트래픽 모델 (Task Arrival):** * 큐잉 이론(Queuing Theory)의 표준인 **푸아송 분포(Poisson Distribution)**를 채택.
  * `numpy.random.poisson(lam)`을 활용하여 매 스텝 버스트(Bursty)한 트래픽 환경을 시뮬레이터 내부에 구현.

## 2. 마르코프 결정 과정 (MDP) 설계

### $\mathcal{S}$: 상태 공간 (State Space)
에이전트(엣지 오케스트레이터)가 매 스텝 관측하는 시스템의 현재 상황입니다. **총 72개의 이산적(Discrete) 상태**로 구성됩니다. ($3 \times 2 \times 2 \times 3 \times 2 = 72$)

| 분류           | 상태 변수 (State Variable)       | 이산화 구간 (Levels) | 경우의 수 |
| :----------- | :--------------------------- | :-------------- | :---: |
| **네트워크**     | 무선 채널 상태 (CQI)               | 나쁨, 보통, 좋음      |   3   |
| **네트워크**     | 가용 대역폭 (Available Bandwidth) | 부족, 여유          |   2   |
| **네트워크**     | 간섭 수준 (Interference Level)   | 낮음, 높음          |   2   |
| **서버 (MDC)** | 서버 CPU 부하량 (Server Workload) | 낮음, 보통, 과부하     |   3   |
| **서버 (MDC)** | 서버 대기열 크기 (Queue Length)     | 즉시 처리 가능, 대기 필요 |   2   |

### $\mathcal{A}$: 행동 공간 (Action Space)
현재 상태에서 에이전트가 내릴 수 있는 자원 할당 결정입니다. 프로젝트 고도화 단계에 따라 두 가지 옵션 중 하나를 선택하여 적용합니다.

* **옵션 A (서버 선택만 집중 / 총 4개 Action):** * 엣지 1로 전송, 엣지 2로 전송, 엣지 3로 전송, 대기/드롭
* **옵션 B (서버 선택 + 대역폭 할당 조인트 / 총 7개 Action):** * 엣지 1 (대역폭 작게/크게), 엣지 2 (대역폭 작게/크게), 엣지 3 (대역폭 작게/크게), 대기/드롭

### $\mathcal{R}$: 보상 함수 (Reward Function)
지연 시간을 최소화하고 통신 장애(Drop)를 회피하도록 유도하는 가중 합산 모델입니다.

$$r_t = - \left( \alpha D_{trans} + \beta D_{comp} + \gamma C_{penalty} \right)$$

* $D_{trans}$: 전송 지연 시간 (채널 상태 및 대역폭 할당량에 비례)
* $D_{comp}$: 연산 지연 시간 (서버 부하 및 대기열 길이에 비례)
* $C_{penalty}$: 큐(Queue) 용량 초과로 인한 패킷 드롭 발생 또는 대역폭 제한 위반 시 부여되는 강력한 벌점 (평소에는 0)
* $\alpha, \beta, \gamma$: 각 항목의 중요도를 결정하는 가중치

## 3. 환경 시뮬레이터 (Environment) 핵심 로직

OpenAI Gym 스타일의 환경으로 구현하며, 내부적으로 통신망의 물리 법칙과 트래픽 발생 무작위성을 처리합니다.

1. **상태 초기화 (`reset`):** 에피소드 시작 시 대기열 비움, 각종 자원 초기 상태 반환.
2. **트래픽 발생 (`step` 내부):** 푸아송 분포로 새 작업 생성 후 큐(Queue)에 적재.
3. **제약 조건 확인 (`step` 내부):** 큐의 최대 용량(`MAX_QUEUE_CAPACITY`)을 넘친 태스크는 Drop 처리하고 $C_{penalty}$ 부여.
4. **결과 반환 (`step` 내부):** 에이전트의 Action을 반영하여 계산된 최종 보상($r_t$)과 업데이트된 다음 상태($s_{t+1}$) 반환.

## 4. SARSA 알고리즘 적용 및 성능 예상

* **사용 기법:** Model-free RL 중 하나인 **Tabular SARSA (On-policy)**
* **업데이트 수식:** $$Q(s_t, a_t) \leftarrow Q(s_t, a_t) + \alpha \left[ r_{t+1} + \gamma Q(s_{t+1}, a_{t+1}) - Q(s_t, a_t) \right]$$
* **성능 및 연산량:** * State 72개와 Action 7개로 이루어진 **504칸($72 \times 7$)의 2차원 행렬(Q-Table)**만 유지하면 됩니다.
  * 차원의 저주를 완벽하게 피한 가벼운 모델링이므로, 딥러닝(Deep Learning) 없이도 1만~5만 스텝 내외(수 초 이내)에 최적 정책으로 초고속 수렴이 가능합니다.
